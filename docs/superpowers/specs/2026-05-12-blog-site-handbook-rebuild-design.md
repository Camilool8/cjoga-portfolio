# Design: blog-site handbook rebuild

**Date:** 2026-05-12
**Owner:** Jose Camilo Joga Guerrero (`cjoga`)
**Status:** Approved, ready for implementation plan
**Target:** `blog-site/` (Docusaurus 3, deploys to `blog.cjoga.cloud`)

## 1. Purpose & scope

The current `blog-site/` is a Docusaurus 3 site with AI-generated placeholder MDX files across four sections (About, Homelab, Work, Certifications). Nothing in `docs/` is real Camilo content — all placeholder, all to be torn out.

This document defines the structure, voice, technical setup, and execution workflow for rebuilding the handbook from scratch with real, interview-driven content.

**Out of scope:**

- Kubernetes manifests (managed in the separate GitOps repo `~/Kodepull/Repositories/kp-k8s-dev`)
- The main portfolio site at `cjoga.cloud` (separate React app, not touched by this work)
- Migrating old blog posts from Supabase (explicitly decided: no blog, docs only)

## 2. Audience & purpose

The handbook serves three audiences simultaneously:

1. **Camilo himself** — personal space to write what he actually thinks. The "be me" part.
2. **Recruiters / hiring managers** — proof of depth beyond a resume. Specifically, depth that the marketing-style `cjoga.cloud` portfolio doesn't carry.
3. **The wider DevOps/Linux community** — readers who want honest, curated guides (starting with RHCSA) and field-tested notes from a real homelab.

The structure satisfies all three by organizing pages by audience: `/me`, `/engineering`, `/learn`.

## 5. Voice & style

Every page is written to these rules. They apply equally to recruiter-facing and reader-facing pages.

### Voice

- **First-person singular.** "I run", not "we run".
- **Direct, plain words.** No "leverage", "synergies", "robust ecosystem".
- **Sober, not flippant.** Personality shows but doesn't perform. No emoji, no exclamation points unless something is actually surprising.
- **Opinions owned.** "This is wrong" instead of "some might argue". When unsure, say "I'm not sure".
- **Concrete over abstract.** Specific incidents and numbers, never vague claims.

### Page structure

- Opening sentence states the page in one breath. No preamble.
- Headings are nouns or short imperatives. Not questions. Not padded with abstractions.
- One idea per paragraph.
- Code blocks have language tags so Prism highlights correctly.
- Mermaid for architecture / flow diagrams. Screenshots only when mermaid can't represent something (UI, dashboard, actual terminal output).
- Cross-links are explicit. "I cover this in `/engineering/lab/network`", not "as discussed earlier".

### Conventions

- **Frontmatter required on every MDX file**: `title`, `description` (drives SEO + social card). Add `sidebar_label` if it differs from title and `sidebar_position` only where order matters.
- **Dates** in `YYYY-MM-DD` on pages where dates appear (`now`, exam dates in `credentials` and `learn/*`).
- **`<AuthorCard />`** at the bottom of every page under `/me`, `/engineering`, and `/learn/*`. Not on the landing or section index pages.
- **No client names on any public-facing content.** Employer name is fine; the end client is not.
- **No AI-generated body content commits.** Skeleton commits are fine; bodies are interview-driven and human-edited before they land in `main`.
- **Length is honest.** A 200-word page that's complete beats a padded 2000-word one. Pages over ~5,000 words get a Docusaurus table of contents.

### Anti-patterns (what to avoid — reasons we burned the placeholders)

- "In this article…" / "This page covers…" intros.
- Bullet lists where prose reads better.
- Headings with one sentence under them.
- Adjective-only claims: "robust", "scalable", "production-grade", "industry-leading" — banned unless followed by a number that proves it.
- Future-tense promises like "soon I will document X". If it's not written, don't promise it.

## 7. Build & deployment

### Repo boundaries

```
THIS REPO (blog-site/)                kp-k8s-dev (separate GitOps repo)
─────────────────────────             ─────────────────────────
src/, docs/, config                   Deployment / Service /
Dockerfile                            HTTPRoute (Envoy Gateway)
.github/workflows/*  (later)    ─→    references image tag
(builds & pushes image)
```

Manifests are **out of scope** for this repo. Any deploy, ingress, or service change is made in `~/Kodepull/Repositories/kp-k8s-dev`. The legacy `kubernetes/portfolio.yaml` in the parent repo is not modified by this work.

### Dockerfile

Multi-stage, minimal:

1. **Builder**: `node:20-alpine`. Copy `package.json` + `package-lock.json`, `npm ci`, copy source, `npm run build`. Outputs `build/`.
2. **Runtime**: `nginx:alpine`. Copy `build/` to web root. Expose 80. Static site only — no Node runtime in the runtime image.

Nginx config: gzip on, fallback to `index.html` for SPA-style deep links (Docusaurus is fully static, so this is mostly for `/learn` and similar paths). Cloudflared Tunnel terminates TLS upstream, so the container speaks plain HTTP.

### Image registry

Docker Hub at `cjoga/blog-site`. Tags: `latest` + commit SHA. Convention matches the existing `cjoga/portfolio`.

### Local development

No change to the existing flow:

```bash
cd blog-site/
npm install
npm start             # docusaurus start, hot reload on port 3000
npm run build         # static build to build/
npm run serve         # serve the built site locally
```

Local image test:

```bash
docker build -t blog-site:dev .
docker run --rm -p 8080:80 blog-site:dev
```

### CI — open item

CI in GitHub Actions is desired but deferred. Camilo will share his current build/push commands and we'll evaluate whether they fit GitHub Actions on his personal account in a follow-up session. Until then, the build is manual.

### Ingress in the cluster

The handbook is exposed via **Envoy Gateway** (HTTPRoute) inside the K3s homelab, with **Cloudflared Tunnel** as the external entry point. There is no NodePort, LoadBalancer, or Traefik IngressRoute. All HTTPRoute and tunnel configuration lives in `kp-k8s-dev`.

## 8. Workflow & order of operations

### Phase 0 — Structural cleanup (one session, no interview required)

Single pass, single commit (or a small batch of focused commits):

1. Delete the 11 placeholder MDX files and the four old `docs/*` folders.
2. Delete `scripts/migrate-from-supabase.mjs`.
3. Remove `@supabase/supabase-js`, `dotenv`, `slugify`, `turndown` from `package.json`. Run `npm install` to regenerate the lockfile.
4. Delete unused static assets listed in Section 6.
5. Create `docs/me/`, `docs/engineering/lab/`, `docs/engineering/work/`, `docs/learn/` with `_category_.json` per folder.
6. Create skeleton MDX files for every page in the Section 4 plan. Each file has real frontmatter and the heading scaffold; body is a single HTML comment placeholder. No prose content. For `/me/opinions/` and `/engineering/lab/tips-and-gotchas/`, only `index.mdx` is created in Phase 0 — individual essays and tips are added during Phase 1+ as Camilo writes them.
7. Update `sidebars.js`, navbar, footer, tagline, and `onBrokenLinks: throw` in `docusaurus.config.js`.
8. Update `src/pages/index.js` (real copy — landing is recruiter-visible from day one) and `src/components/AuthorCard/index.js` (bio sentence dictated by Camilo).
9. Add the `Dockerfile`.
10. Verify: `npm run build` succeeds with `onBrokenLinks: throw`. `npm start` renders the new structure with skeleton pages visible.
11. Commit.

### Phase 1+ — Interview pass (one page per session, paced by Camilo)

Each page follows the same loop:

```
Camilo & me on a page  →  I ask focused questions
                       →  I draft from his answers
                       →  He corrects/edits
                       →  I revise
                       →  He approves
                       →  Commit page
```

**Per-page verification before commit:**

- Frontmatter is real (`title`, `description`).
- No "TBD", no `<!-- to be filled -->`, no AI hedging.
- Voice matches Section 5.
- `npm run build` succeeds (broken links would fail it).
- Every cited fact is verifiable.

**One page = one commit** so the git history reads as page-by-page, deliberate work.

### Page order

1. `/me/who-i-am`
2. `/me/credentials`
3. `/engineering/work/inspyr-global-solutions`
4. `/engineering/work/arctiq`
5. `/engineering/work/fl-betances`

Rationale: identity → recruiter signal → meaty technical depth → community guides. RHCSA last because it needs the most care and is anchored on an existing repo — gives time to prep without pressure.

## 10. Non-goals

To be unambiguous about what this design does **not** include:

- A blog (`docusaurus.config.js` keeps `blog: false`).
- i18n / Spanish locale (English only).
- A separate "field notes" or "tips" top-level section — tips live inside `/engineering/lab/tips-and-gotchas/`.
- Naming end clients (Deloitte, W.W. Norton, or any other) anywhere on the site.
- Kubernetes manifest changes in this repo.
- Migrating the existing Supabase blog posts.
